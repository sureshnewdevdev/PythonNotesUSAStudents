1. OOP Foundations

Before moving into modern Python topics like dataclasses and async programming, we should refresh the object-oriented foundation. In most corporate projects, OOP helps us organize code into manageable units. Instead of keeping everything in loose functions and variables, we model the business using classes and objects.

What is a Class?

A class is a blueprint. It defines the structure and behavior that objects created from it should have.

What is an Object?

An object is the actual instance created from a class. If Employee is a class, then emp1 and emp2 are objects.

CLASS +----------------------+ | Employee | +----------------------+ | name | | employee_id | | department | +----------------------+ | work() | | display_details() | +----------------------+ | | | creates | creates v v +---------------+ +---------------+ | emp1 | | emp2 | +---------------+ +---------------+ | name = Ravi | | name = Meera | | id = 101 | | id = 102 | +---------------+ +---------------+
Trainer note: Think of a class as an application form format, and an object as a filled application for one specific person.

Basic Example

class Employee:
    def __init__(self, name, employee_id, department):
        self.name = name
        self.employee_id = employee_id
        self.department = department

    def work(self):
        print(f"{self.name} is working in {self.department} department.")

    def display_details(self):
        print("Employee Details")
        print("Name:", self.name)
        print("ID:", self.employee_id)
        print("Department:", self.department)


emp1 = Employee("Ravi", 101, "Finance")
emp2 = Employee("Meera", 102, "Technology")

emp1.work()
emp2.display_details()

How to Read This Example

  • class Employee: creates a class definition.
  • __init__ is the constructor. It runs when an object is created.
  • self refers to the current object.
  • self.name, self.employee_id, and self.department are instance variables.
  • work() and display_details() are methods.
Common beginner confusion: A class is not the same as an object. The class describes. The object lives.

Real-World Corporate Example

Suppose we are building an HR application. We might model employees, managers, payroll processors, and departments as classes. This improves readability and makes the system easier to extend later.

2. Four Pillars of OOP

2.1 Encapsulation

Encapsulation means keeping data and the related behavior together in one unit, and controlling access to sensitive values.

class BankAccount:
    def __init__(self, holder_name, balance):
        self.holder_name = holder_name
        self.__balance = balance   # private-like variable

    def deposit(self, amount):
        if amount > 0:
            self.__balance += amount

    def withdraw(self, amount):
        if 0 < amount <= self.__balance:
            self.__balance -= amount
        else:
            print("Insufficient balance or invalid amount")

    def get_balance(self):
        return self.__balance


account = BankAccount("Anita", 5000)
account.deposit(2000)
account.withdraw(1000)
print("Available Balance:", account.get_balance())
Encapsulation Idea +--------------------------------------+ | BankAccount | +--------------------------------------+ | holder_name | | __balance <-- protected internally | +--------------------------------------+ | deposit() | | withdraw() | | get_balance() | +--------------------------------------+

Here, users do not manipulate __balance directly. They go through methods such as deposit() and withdraw().

2.2 Inheritance

Inheritance allows one class to reuse the features of another class.

class Employee:
    def __init__(self, name):
        self.name = name

    def show_role(self):
        print(f"{self.name} is an employee")


class Manager(Employee):
    def approve_leave(self):
        print(f"{self.name} can approve leave requests")


mgr = Manager("Suresh")
mgr.show_role()
mgr.approve_leave()
+----------------+ | Employee | +----------------+ | name | +----------------+ | show_role() | +----------------+ ^ | | +----------------+ | Manager | +----------------+ | approve_leave()| +----------------+

The Manager class inherits from Employee, so it gets show_role() without rewriting it.

2.3 Polymorphism

Polymorphism means one interface, many forms. The same method name can behave differently depending on the object.

class EmailNotifier:
    def send(self):
        print("Sending email notification")


class SMSNotifier:
    def send(self):
        print("Sending SMS notification")


def trigger_notification(notifier):
    notifier.send()


trigger_notification(EmailNotifier())
trigger_notification(SMSNotifier())

The function trigger_notification() works with any object that provides a send() method.

2.4 Abstraction

Abstraction means exposing only the essential features and hiding complex implementation details.

from abc import ABC, abstractmethod

class PaymentProcessor(ABC):
    @abstractmethod
    def pay(self, amount):
        pass


class CreditCardPayment(PaymentProcessor):
    def pay(self, amount):
        print(f"Paid {amount} using credit card")


class UpiPayment(PaymentProcessor):
    def pay(self, amount):
        print(f"Paid {amount} using UPI")


payment1 = CreditCardPayment()
payment2 = UpiPayment()

payment1.pay(2500)
payment2.pay(900)
Why abstraction matters in projects: Your application can depend on a common contract like PaymentProcessor without caring about whether the actual payment happens through UPI, card, or net banking.

3. Dataclasses

In many applications, we create classes mainly to store data. Writing constructors, string methods, and comparisons repeatedly becomes tedious. Python provides dataclasses to reduce that boilerplate.

Without Dataclass

class Product:
    def __init__(self, product_id, name, price):
        self.product_id = product_id
        self.name = name
        self.price = price

    def __repr__(self):
        return f"Product(product_id={self.product_id}, name='{self.name}', price={self.price})"

With Dataclass

from dataclasses import dataclass

@dataclass
class Product:
    product_id: int
    name: str
    price: float


p1 = Product(101, "Laptop", 55000.0)
print(p1)

The @dataclass decorator automatically generates useful methods like __init__, __repr__, and sometimes __eq__.

Detailed Example with Business Logic

from dataclasses import dataclass, field

@dataclass
class EmployeeRecord:
    employee_id: int
    name: str
    basic_salary: float
    allowances: float = 0.0
    deductions: float = 0.0
    department: str = "General"
    email: str = field(default="not-set@company.com")

    def net_salary(self) -> float:
        return self.basic_salary + self.allowances - self.deductions


emp = EmployeeRecord(
    employee_id=1001,
    name="Karthik",
    basic_salary=50000,
    allowances=5000,
    deductions=2000,
    department="IT",
    email="karthik@company.com"
)

print(emp)
print("Net Salary:", emp.net_salary())

Why Dataclasses Are Useful

  • Less boilerplate code
  • Cleaner models for API requests/responses
  • Useful for configuration objects and DTO-like structures
  • Easy readability for teams

Advanced Dataclass Features

FeatureMeaningExample Idea
defaultSet a fixed default valueStatus = "Active"
default_factoryGenerate a new default value each timeEmpty list per object
frozen=TrueMakes object immutable-likeSafe config object
order=TrueAdds comparison supportSort records by fields
from dataclasses import dataclass, field

@dataclass(frozen=True)
class AppConfig:
    app_name: str
    supported_regions: list[str] = field(default_factory=list)

config = AppConfig("PayrollSystem", ["India", "UAE"])
print(config)
Important: Use default_factory for mutable defaults like lists and dictionaries. Do not write items=[] directly in dataclass fields.

4. Typing in Python

Python is dynamically typed, but modern Python projects strongly benefit from type hints. They improve readability, IDE support, refactoring, and team communication.

Basic Type Hints

def calculate_bonus(salary: float, percentage: float) -> float:
    return salary * percentage / 100


bonus = calculate_bonus(60000, 10)
print(bonus)

Common Type Hint Patterns

from typing import List, Dict, Tuple, Optional

def get_employee_names() -> List[str]:
    return ["Asha", "Rohan", "Divya"]

def get_department_map() -> Dict[int, str]:
    return {101: "HR", 102: "IT"}

def get_coordinates() -> Tuple[float, float]:
    return (12.97, 77.59)

def find_manager(employee_id: int) -> Optional[str]:
    managers = {1: "Prakash"}
    return managers.get(employee_id)

Using Modern Built-In Generic Syntax

def get_sales() -> list[int]:
    return [100, 200, 300]

def get_settings() -> dict[str, str]:
    return {"mode": "production", "region": "india"}

Type Aliases

type EmployeeData = dict[str, str | int]

def print_employee(data: EmployeeData) -> None:
    print(data)

Protocols and Duck Typing

In real Python design, you often care more about what an object can do than what exact class it belongs to.

from typing import Protocol

class Printable(Protocol):
    def print_report(self) -> None:
        ...

class SalesReport:
    def print_report(self) -> None:
        print("Printing sales report")

def export_report(obj: Printable) -> None:
    obj.print_report()

export_report(SalesReport())
Trainer tip: Typing is not just for compilers. It is also for humans. When your teammate opens your code after six months, type hints reduce confusion immediately.

5. Async Patterns

Asynchronous programming helps when your application spends time waiting for external work such as APIs, database calls, or file/network operations. While one task is waiting, another task can continue.

Synchronous Style Task A ----wait----finish Task B ----wait----finish Task C ----wait----finish Total time = A + B + C Asynchronous Style Task A ----wait-----------finish Task B ----wait------finish Task C ----wait----------------finish Total time can be closer to the longest task, not the sum of all tasks

Basic async / await Example

import asyncio

async def fetch_data(source: str) -> str:
    print(f"Fetching from {source}...")
    await asyncio.sleep(2)
    return f"Data from {source}"

async def main():
    result1 = await fetch_data("API-1")
    result2 = await fetch_data("API-2")
    print(result1)
    print(result2)

asyncio.run(main())

This example is async, but still sequential because each call is awaited one after another.

Running Tasks Concurrently

import asyncio

async def fetch_data(source: str) -> str:
    print(f"Start fetching from {source}")
    await asyncio.sleep(2)
    return f"Completed {source}"

async def main():
    task1 = asyncio.create_task(fetch_data("Service-A"))
    task2 = asyncio.create_task(fetch_data("Service-B"))
    task3 = asyncio.create_task(fetch_data("Service-C"))

    results = await asyncio.gather(task1, task2, task3)
    for result in results:
        print(result)

asyncio.run(main())

When to Use Async

  • Calling multiple APIs
  • Reading from multiple services
  • Web scraping or concurrent network operations
  • Async web frameworks like FastAPI or aiohttp-based code

When Not to Use Async

  • Pure CPU-heavy work such as large mathematical processing
  • Very small scripts where complexity is not worth it
  • Codebases where the libraries you use are fully synchronous

Producer-Consumer Pattern

import asyncio

async def producer(queue: asyncio.Queue):
    for item in range(1, 6):
        await asyncio.sleep(1)
        await queue.put(item)
        print(f"Produced: {item}")
    await queue.put(None)

async def consumer(queue: asyncio.Queue):
    while True:
        item = await queue.get()
        if item is None:
            break
        print(f"Consumed: {item}")

async def main():
    queue = asyncio.Queue()
    await asyncio.gather(producer(queue), consumer(queue))

asyncio.run(main())
Corporate use case: A producer can collect incoming messages, and a consumer can process them for notifications, logging, or transformations.

6. Packaging for Reuse

As your Python code grows, you should avoid keeping everything in one file. Packaging helps you organize modules and reuse your code across projects.

Module vs Package

TermMeaning
ModuleA single Python file like salary_utils.py
PackageA folder containing Python modules, usually with __init__.py
payroll_app/ | +-- main.py +-- payroll/ | +-- __init__.py | +-- calculations.py | +-- validators.py | +-- reports.py | +-- tests/ +-- test_calculations.py

Example Files

# payroll/calculations.py
def calculate_net_salary(basic: float, allowance: float, deduction: float) -> float:
    return basic + allowance - deduction
# payroll/validators.py
def is_valid_salary(amount: float) -> bool:
    return amount >= 0
# main.py
from payroll.calculations import calculate_net_salary
from payroll.validators import is_valid_salary

salary = calculate_net_salary(50000, 4000, 2000)

if is_valid_salary(salary):
    print("Net salary:", salary)

Why Packaging Matters

  • Improves maintainability
  • Makes code reusable across multiple projects
  • Supports testing and versioning
  • Encourages clean separation of concerns

Basic pyproject.toml Example

[project]
name = "payroll-utils"
version = "0.1.0"
description = "Reusable payroll helper utilities"
readme = "README.md"
requires-python = ">=3.11"

[build-system]
requires = ["setuptools>=61.0"]
build-backend = "setuptools.build_meta"

Typical Reuse Strategy in Teams

  1. Create a clean package structure.
  2. Keep domain models separate from utility helpers.
  3. Add type hints and tests.
  4. Publish internally or reuse via Git repository / artifact management.
Trainer insight: If three applications repeat the same validation or logging logic, that code is a candidate for packaging and reuse.

7. Best Practices

For OOP

  • Prefer meaningful class names
  • Keep classes focused on one responsibility
  • Use inheritance only where an “is-a” relationship exists
  • Prefer composition when suitable

For Dataclasses

  • Use for data-heavy models
  • Use default_factory for mutable fields
  • Add methods only when they add domain value

For Typing

  • Add return types to public functions
  • Use modern generic syntax when possible
  • Keep type hints readable, not overly complex

For Async

  • Use for I/O-bound tasks
  • Avoid blocking calls inside async functions
  • Use asyncio.gather() carefully and handle failures

8. MCQs

1. What is the main purpose of a class in OOP?

A. To store only one value
B. To act as a blueprint for objects
C. To replace functions completely
D. To avoid variables

Correct answer: B. A class acts as a blueprint from which objects are created.
2. Which OOP concept is demonstrated when a child class reuses a parent class?

A. Abstraction
B. Encapsulation
C. Inheritance
D. Compilation

Correct answer: C. Inheritance enables reuse of existing behavior from a base class.
3. Why do we use @dataclass?

A. To make Python static
B. To automatically generate boilerplate methods
C. To block object creation
D. To remove methods from classes

Correct answer: B. Dataclasses reduce repetitive code such as constructors and representation methods.
4. Which keyword is used to pause inside an async function until a task completes?

A. yield
B. pause
C. break
D. await

Correct answer: D. await is used with awaitable operations inside async functions.
5. What is a package in Python?

A. A comment block
B. A database table
C. A folder containing Python modules

Correct answer: C. A package is a directory structure used to organize related modules.

9. Interview Questions and Answers

Q1. Why is Python considered object-oriented even though it is dynamically typed?
Python supports classes, objects, inheritance, polymorphism, and encapsulation. Dynamic typing means variable types are resolved at runtime, but that does not reduce Python’s object-oriented capabilities.
Q2. When would you prefer a dataclass over a normal class?
Use a dataclass when the class is mainly used for storing and transporting data. It reduces boilerplate and makes models cleaner, especially in service layers, configurations, API payloads, and DTO-style objects.
Q3. What problem do type hints solve in real projects?
Type hints improve readability, tooling support, refactoring confidence, and team communication. They reduce ambiguity and make larger codebases easier to maintain.
Q4. Explain one scenario where async programming is useful.
Async programming is useful when the application makes multiple network calls, such as fetching data from several microservices. Instead of waiting for each call one by one, the program can handle them concurrently and reduce total waiting time.
Q5. What is the benefit of packaging code for reuse?
Packaging improves organization, reusability, testing, versioning, and maintainability. Shared logic can be reused across projects rather than duplicated again and again.
